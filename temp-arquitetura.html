1. Visão Geral da Solução

Esta solução implementa uma plataforma de orquestração de processos de negócio utilizando o Temporal como motor de workflow determinístico, integrado com capacidades de Inteligência Artificial em pontos específicos da execução. Toda a infraestrutura reside em uma única aplicação que inicializa três workers, cada um escutando uma task queue dedicada.

O princípio central é a separação entre orquestração e execução. O Temporal controla o fluxo, mantém o estado e garante a resiliência. A IA opera exclusivamente dentro de atividades específicas, nunca no nível da orquestração.

1.1 Princípios Arquiteturais

  • Aplicação única: os três workers rodam no mesmo processo, escutando filas distintas. Escala independente pode ser obtida futuramente separando-os.
  • Determinismo na orquestração: toda decisão sobre fluxo é tomada em código, sem IA.
  • Registro de atividades: o worker de automação mantém um registro de implementações disponíveis. O Markdown referencia qual usar em cada etapa. Trocar a implementação não exige alterar o worker.
  • IA como detalhe de implementação: utilizada dentro de atividades registradas, nunca para decidir o próximo passo do processo.
  • Estado centralizado no Temporal: os resultados de todas as etapas são armazenados e recuperados diretamente pelo Temporal.
  • Flexibilidade via Notion: definições de processo e regras de navegação vivem em Markdown no Notion, não no código.
  • Simetria humano/agente: atividades com humano no loop e atividades com agente de IA seguem o mesmo padrão de execução via MCP.
  • Etapas como unidade universal: notificações, webhooks e execuções são todas etapas do processo — definidas no Markdown, rastreadas pelo Temporal.

2. Notion Markdown — Definição de Processo

O Markdown armazenado no Notion é o elemento central de flexibilidade da solução. Cada processo de negócio é definido em um documento com schema bem definido. O worker de orquestração lê esse documento para entender as etapas, os tipos, as implementações referenciadas e as regras de navegação entre elas.

Princípio fundamental: o código dos workers nunca muda para acomodar novos processos. Toda a lógica específica de cada processo vive no Markdown do Notion. A estrutura completa do schema é documentada em artefato separado.

2.1 Tipos de Etapa

Cada etapa declara um tipo no Markdown. O tipo determina para qual fila o worker de orquestração roteia a atividade e qual comportamento o worker de execução adota.

Tipo

Fila de destino / Comportamento

hitl_humano

hitl-queue. Worker aguarda interação via MCP iniciada pelo humano.

hitl_agente

hitl-queue. Worker aguarda interação via MCP iniciada pelo agente de IA (acordado por webhook na etapa anterior).

webhook

automation-queue. Worker executa disparo de webhook nativamente. Sem necessidade de implementação registrada adicional.

automatizada

automation-queue. Worker consulta o campo atividade da etapa e executa a implementação registrada correspondente.

2.2 O Campo 'atividade' nas Etapas Automatizadas

Etapas do tipo automatizada referenciam uma implementação específica registrada no worker de automação através do campo atividade. Esse campo é o contrato entre o Markdown e o worker.

Exemplos de referências válidas:

Valor do campo 'atividade'

Implementação no worker

analisar_documento_ia

Chama API de IA com contexto do Temporal e Markdown do processo.

analisar_documento_codigo

Executa lógica determinística em código puro.

calcular_imposto

Executa cálculo determinístico baseado nos parâmetros da etapa.

gerar_relatorio

Gera documento com template fixo e dados do Temporal.

notificar_humano

Envia notificação via canal configurado. (Implementação futura.)

Para substituir uma implementação com IA por código determinístico — ou o inverso — basta alterar o valor do campo atividade no Markdown do processo. Desde que a implementação de destino já esteja registrada no worker, nenhuma linha de código precisa ser alterada.

O worker de automação cresce por adição, nunca por modificação. Novas implementações são registradas uma vez. A partir daí, qualquer processo pode referenciá-las no Markdown.

2.3 Regras de Navegação

As regras de navegação determinam, para cada etapa, qual é a próxima a ser executada. Avaliadas pelo worker de orquestração em código puro, com base no resultado da atividade anterior armazenado no Temporal.

  • Navegação simples: após etapa A, executar etapa B.
  • Navegação condicional: se resultado da etapa A for X, executar B; se for Y, executar C.
  • Loop: retornar à etapa B até que determinada condição seja atendida.
  • Retorno: voltar a uma etapa anterior com base em critério específico.
  • Finalização: encerrar o processo após determinada etapa.

2.4 Padrão de Etapas Antecedentes

Antes de qualquer atividade que aguarda um ator externo, o processo define etapas antecedentes de aviso no próprio Markdown. Essas etapas são rastreadas pelo Temporal como qualquer outra atividade.

Atividade principal

Etapa antecedente recomendada

hitl_humano — aguarda humano via MCP

Etapa automatizada com atividade notificar_humano. (Implementação futura. Por ora, humano consulta ativamente o agente.)

hitl_agente — aguarda agente IA via MCP

Etapa do tipo webhook. Nativa no worker — acorda o agente, que vem via MCP executar a atividade.

3. Worker de Orquestração

O worker de orquestração é o cérebro determinístico da solução. Recebe um process_id, localiza o Markdown no Notion, e gerencia o ciclo de vida completo do processo: avalia resultados, aplica regras de navegação e enfileira a próxima atividade na fila correta.

Este worker é compilado uma única vez e nunca precisa ser alterado. Toda a especificidade de cada processo está no Markdown do Notion.

3.1 Fluxo de Execução

1

Recebe o process_id como entrada do workflow Temporal (via orchestration-queue).

2

Consulta o Notion e recupera o Markdown correspondente ao process_id.

3

Faz o parse do Markdown: identifica etapas, tipos, implementações referenciadas e regras de navegação.

4

Identifica a primeira etapa e a enfileira na fila correta com base no tipo declarado.

5

Aguarda a conclusão da atividade. O Temporal bloqueia o workflow até o sinal de conclusão.

6

Recupera o resultado da atividade concluída, armazenado no Temporal.

7

Avalia o resultado contra as regras de navegação do Markdown — em código, sem IA.

8

Decide a próxima etapa e a enfileira na fila correta, passando tipo, implementação referenciada e parâmetros.

9

Repete os passos 5 a 8 até que uma regra de finalização encerre o processo.

3.2 Roteamento por Tipo de Etapa

  • Tipos hitl_humano e hitl_agente → enfileirados em hitl-queue.
  • Tipos webhook e automatizada → enfileirados em automation-queue.

3.3 O que o Worker de Orquestração NÃO faz

  • Não executa nenhuma atividade de negócio diretamente.
  • Não utiliza IA em nenhuma etapa de decisão.
  • Não mantém estado fora do Temporal.
  • Não é alterado quando um novo processo é criado ou modificado.

4. Worker HITL — Human-in-the-Loop

O worker HITL gerencia atividades que requerem interação de um ator externo via MCP — seja um humano em uma sessão Claude.ai ou um agente de IA acordado por webhook. O padrão de interação é idêntico nos dois casos.

O mesmo contrato MCP serve humanos e agentes de IA. A diferença está em quem inicia a interação e quando: o humano age quando quiser; o agente age assim que recebe o webhook disparado pela etapa antecedente.

4.1 Fluxo — Humano

1

Worker de orquestração enfileira etapa de notificação (futura) seguida de etapa hitl_humano em hitl-queue.

2

Humano abre sessão com o agente (Claude.ai) e pergunta o que está pendente.

3

Agente chama list_hitl_activities() e apresenta a lista ao humano.

4

Humano escolhe a atividade que deseja trabalhar.

5

Agente chama start_activity(id). Recebe contexto completo: histórico de etapas anteriores + Markdown do processo.

6

Humano e agente trabalham juntos para resolver a atividade.

7

Agente chama complete_activity(id, result). Temporal registra o resultado e sinaliza conclusão ao worker de orquestração.

4.2 Fluxo — Agente de IA

1

Worker de orquestração enfileira etapa de webhook (automation-queue) seguida de etapa hitl_agente (hitl-queue).

2

Worker de automação dispara o webhook para o endpoint do agente, passando o id da atividade.

3

Agente acorda ao receber o webhook. Já possui o id — não precisa listar atividades.

4

Agente chama start_activity(id). Recebe contexto completo: histórico de etapas anteriores + Markdown do processo.

5

Agente executa a atividade de forma autônoma, com base no contexto recebido.

6

Agente chama complete_activity(id, result). Temporal registra o resultado e sinaliza conclusão ao worker de orquestração.

4.3 Métodos MCP

Método

Comportamento

list_hitl_activities()

Retorna atividades pendentes em hitl-queue com metadados suficientes para o humano escolher. Não utilizado pelo agente de IA — ele já recebe o id via webhook.

start_activity(id)

Marca atividade como iniciada. Retorna contexto completo: resultados das etapas anteriores (Temporal) + Markdown do processo. Ponto de entrada para ambos os atores.

complete_activity(id, result)

Registra resultado no Temporal e sinaliza conclusão. Worker de orquestração recebe o sinal e avalia a próxima etapa.

5. Worker de Automação

O worker de automação executa atividades que não requerem interação humana direta. Ele mantém um registro de implementações disponíveis e executa a implementação correta com base no campo atividade declarado no Markdown da etapa. O disparo de webhook é uma capacidade nativa — não requer implementação registrada adicional.

5.1 Registro de Implementações

O worker mantém um mapa de identificadores para funções de execução. Cada implementação é registrada uma vez. A partir daí, qualquer processo pode referenciá-la no Markdown sem nenhuma alteração no código do worker.

  • O registro cresce por adição. Implementações existentes nunca são modificadas para acomodar novos processos.
  • Uma implementação pode ser determinística (código puro) ou baseada em IA (chamada interna à API). Do ponto de vista do worker e do Temporal, ambas são equivalentes: recebem parâmetros e retornam resultado.
  • Substituir uma implementação com IA por código determinístico exige apenas: (1) registrar a nova implementação no worker, (2) atualizar o campo atividade no Markdown do processo. Sem deploy adicional se a implementação já estiver registrada.

A IA nas implementações do tipo automatizada é um detalhe interno. Ela não é exposta via MCP. O Temporal vê apenas: atividade iniciada → atividade concluída com resultado.

5.2 Implementações Nativas

Algumas implementações fazem parte da aplicação desde o início, sem necessidade de desenvolvimento adicional:

Identificador nativo

Comportamento

webhook

Dispara requisição HTTP para URL configurada no Markdown. Parâmetros: url, payload, método, autenticação. Resultado: resposta do endpoint. Retry automático pelo Temporal em caso de falha.

5.4 Contexto nas Implementações com IA

Toda implementação que aciona IA internamente recebe obrigatoriamente dois elementos de contexto, garantindo execução altamente contextualizada:

  • Resultados de todas as etapas anteriores do processo, recuperados diretamente do Temporal.
  • Markdown completo do processo no Notion, incluindo instruções específicas da etapa em execução.

6. Gerenciamento de Estado e Contexto

O Temporal é a única fonte de verdade para o estado de cada processo em execução. Não há banco de dados externo, cache ou arquivo intermediário para armazenar contexto entre etapas.

6.1 Fluxo de Estado

  • Cada atividade concluída registra seu resultado no Temporal.
  • O worker de orquestração acessa esses resultados ao avaliar as regras de navegação.
  • O start_activity recupera o histórico de resultados do Temporal e o entrega ao ator (humano ou agente).
  • Implementações com IA no worker de automação recuperam o histórico antes de acionar a API.
  • O complete_activity registra o novo resultado, disponível para as etapas seguintes.

6.2 Vantagens

  • Simplicidade operacional: um único sistema gerencia estado, histórico e resiliência.
  • Rastreabilidade completa: todo o histórico de execução fica registrado nativamente.
  • Resiliência: o Temporal garante reexecução de atividades em caso de falha, com estado preservado.
  • Sem dependências adicionais: não há necessidade de Redis, banco relacional ou sistema externo para contexto.

7.2 Roadmap de Evolução

  • Fase 1 — MVP: aplicação única com três workers, webhook nativo, fluxo HITL humano (pull), fluxo HITL agente via webhook, primeiras implementações registradas.
  • Fase 2 — Notificação humana: implementar e registrar a atividade notificar_humano no worker de automação.
  • Fase 3 — Observabilidade: dashboard de processos em execução, histórico de atividades e métricas.
  • Fase 4 — Dry-run: validação de Markdown de processo antes de ir para produção.
  • Fase 5 — Escala independente: separação dos workers em aplicações distintas se volume exigir.

8. Resumo Executivo

Esta arquitetura entrega uma plataforma de orquestração de processos que combina a confiabilidade do Temporal com a flexibilidade da IA. O código é genérico e estável; a lógica de negócio vive no Notion. Novos processos são criados sem alterar uma linha de código.

A solução roda em uma aplicação única com três workers. O worker de orquestração nunca muda. O worker de automação cresce por adição de implementações registradas — determinísticas ou com IA. O worker HITL serve humanos e agentes pelo mesmo contrato MCP.

Trocar uma atividade de IA para código determinístico é uma operação de Markdown: registra-se a implementação no worker uma vez, e o Markdown do processo passa a referenciar o novo identificador. O Temporal une tudo como fonte única de estado, com cada atividade rastreada, auditável e resiliente.
